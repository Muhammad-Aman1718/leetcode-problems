1. REST - Sabse Popular API

üì¶ Data Format

Primary Format: JSON (99% cases)
Alternative Formats: XML HTML Plain Text

JSON ko prefer karte hain kyunki ye lightweight aur readable hai.

// REST Request Example (JSON) POST /api/users HTTP/1.1 Host: example.com Content-Type: application/json { "name": "Ali Ahmed", "email": "ali@example.com", "age": 25 } // REST Response (JSON) { "id": 123, "name": "Ali Ahmed", "status": "success" }

üîß HTTP Methods (CRUD Operations)

POST
Naya data create

PATCH
Partial update
DELETE
Data delete karna
HEAD

OPTIONS
Methods check karna

üè¢ Famous Companies Using REST

Twitter API Facebook Graph API Instagram API Google Maps API Amazon AWS PayPal Stripe GitHub API Spotify API YouTube API
üöÄ Transport Protocol: HTTP/HTTPS (Port 80/443)

2. SOAP - Enterprise Level Protocol

üì¶ Data Format

Only Format: XML

SOAP sirf XML use karta hai - koi choice nahi. Bohot verbose (lambi) hoti hai.

<!-- SOAP Request Example (XML) --> <soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/"> <soap:Header/> <soap:Body> <GetUserDetails> <UserId>123</UserId> </GetUserDetails> </soap:Body> </soap:Envelope> <!-- SOAP Response --> <soap:Envelope> <soap:Body> <GetUserDetailsResponse> <User> <Name>Ali Ahmed</Name> <Email>ali@example.com</Email> </User> </GetUserDetailsResponse> </soap:Body> </soap:Envelope>

üîß Operations (REST jaise methods nahi)

SOAP mein HTTP methods ka concept nahi hai!

Ye function/method based hai. Har operation ka alag function hota hai:
  • GetUser - User details lena
  • CreateUser - Naya user banana
  • UpdateUser - User update karna
  • DeleteUser - User delete karna
‚ö†Ô∏è Sab operations POST request mein jaate hain! GET, PUT, DELETE nahi use hote.

üè¢ Famous Companies Using SOAP

Salesforce API PayPal (Old APIs) Microsoft .NET Services Banks (HBL, UBL, etc.) Government Systems IBM WebServices Oracle Apps SAP Systems
üöÄ Transport Protocol: HTTP, HTTPS, SMTP, TCP, JMS (multiple options)

3. gRPC - Google's Speed Monster

üì¶ Data Format

Format: Protocol Buffers (Protobuf)
Type: Binary (Human-readable nahi)

JSON se 3-10x chhota aur tez! Binary format hai to directly nahi dekh sakte.

// Protocol Buffer Definition (.proto file) syntax = "proto3"; message User { int32 id = 1; string name = 2; string email = 3; int32 age = 4; } service UserService { rpc GetUser(UserId) returns (User); rpc CreateUser(User) returns (User); rpc UpdateUser(User) returns (User); rpc DeleteUser(UserId) returns (Empty); } // Binary format mein convert ho kar jata hai // Human readable nahi hota!

üîß RPC Methods (Remote Procedure Calls)

gRPC mein HTTP methods nahi, RPC calls hote hain:

Unary RPC
Simple request-response
Server Streaming
Server continuous data bheje
Client Streaming
Client continuous data bheje
Bidirectional
Dono taraf streaming
üí° gRPC internally HTTP/2 use karta hai jo multiplexing support karta hai!

üè¢ Famous Companies Using gRPC

Netflix Google (Internal) Uber Square CoreOS Cisco Dropbox Docker
üöÄ Transport Protocol: HTTP/2 (Binary Protocol)

4. GraphQL - Flexible Query Language

üì¶ Data Format

Request Format: GraphQL Query Language
Response Format: JSON

Request mein special GraphQL syntax, response JSON mein milta hai.

# GraphQL Query (Request) query { user(id: 123) { name email posts { title likes } } } // GraphQL Response (JSON) { "data": { "user": { "name": "Ali Ahmed", "email": "ali@example.com", "posts": [ { "title": "My First Post", "likes": 100 } ] } } }

üîß Operations (3 Types)

GraphQL mein sirf 3 operations hain:

Query
Data read (GET jaise)
Mutation
Data create/update/delete (POST/PUT/DELETE jaise)
Subscription
Real-time updates (WebSocket jaise)
üí° Sab operations ek hi endpoint pe jaate hain: /graphql
Mostly POST request use hoti hai (query bhi POST mein ja sakti hai)

üè¢ Famous Companies Using GraphQL

Facebook GitHub Shopify Twitter (New API) Airbnb Netflix (Falcor - similar) PayPal The New York Times
üöÄ Transport Protocol: HTTP/HTTPS (Usually POST), WebSocket (for subscriptions)

5. WebRTC - Peer-to-Peer Communication

üì¶ Data Format

Signaling Phase: JSON (SDP format)
Media Data: Binary Streams (RTP packets)
Data Channel: Any format (Binary ya Text)

Video/Audio binary streams mein jaata hai, control messages JSON mein.

// WebRTC Signaling (JSON) { "type": "offer", "sdp": "v=0\r\no=- 123 2 IN IP4 127.0.0.1\r\n..." } // ICE Candidate (JSON) { "candidate": "candidate:1 1 UDP 2130706431...", "sdpMid": "0", "sdpMLineIndex": 0 } // Actual video/audio: Binary RTP packets // Direct peer-to-peer stream - server se nahi guzarta

üîß Communication Methods

WebRTC mein HTTP methods nahi hain - ye peer-to-peer hai!

  • Signaling: Connection setup (usually WebSocket ya HTTP)
  • STUN: Public IP discover karne ke liye
  • TURN: Relay server (jab direct connection fail ho)
  • RTP/SRTP: Actual media streaming
  • Data Channels: Custom data transfer

üè¢ Famous Companies Using WebRTC

Google Meet Zoom WhatsApp Web Discord Facebook Messenger Skype Microsoft Teams Snapchat
üöÄ Transport Protocol: UDP (SRTP/SRTCP), ICE, STUN, TURN

6. Webhooks - Event-Driven Callbacks

üì¶ Data Format

Primary Format: JSON
Alternative: Form-urlencoded XML

99% webhooks JSON use karte hain. Server automatically POST request bhejta hai.

// Webhook Payload Example (Payment Success) POST /webhook/payment HTTP/1.1 Host: yourserver.com Content-Type: application/json { "event": "payment.success", "data": { "transaction_id": "TXN12345", "amount": 5000, "currency": "PKR", "customer": "ali@example.com" }, "timestamp": "2024-01-15T10:30:00Z" }

üîß Methods

Webhooks mein sirf POST use hota hai!

POST
Server automatically tumhare endpoint pe POST request bhejta hai jab event occur ho
üí° Tum request nahi karte, server automatic notification bhejta hai!
Tumhe bas endpoint provide karna hota hai jahan server hit karega.

üè¢ Famous Companies Using Webhooks

Stripe (Payments) PayPal GitHub (Push events) Slack (Bot events) Shopify Twilio SendGrid Mailchimp
üöÄ Transport Protocol: HTTP/HTTPS (POST requests only)

7. WebSockets - Persistent Connection

üì¶ Data Format

Common Formats: JSON Plain Text Binary
Flexibility: Koi bhi format bhej sakte ho!

JSON sabse common hai, lekin binary data bhi bhej sakte ho (images, files, etc.)

// WebSocket Connection Establishment GET /chat HTTP/1.1 Host: example.com Upgrade: websocket Connection: Upgrade // WebSocket Message (JSON) { "type": "message", "user": "Ali", "text": "Hello everyone!", "timestamp": "2024-01-15T10:30:00Z" } // Plain Text Message "Ali: Hello everyone!" // Binary Message [Binary image data...]

üîß Communication Pattern

WebSockets mein HTTP methods nahi - bidirectional messaging!

  • send(): Client se server ko message
  • onmessage: Server se message receive
  • Bidirectional: Dono taraf kabhi bhi message bhej sakte hain
  • Full-duplex: Same time pe send aur receive
üí° Connection ek baar establish ho jaye to HTTP khatam - ab sirf WebSocket frames!

üè¢ Famous Companies Using WebSockets

WhatsApp Web Slack Trello Binance (Trading) Trading Platforms Online Games Figma Google Docs
üöÄ Transport Protocol: WebSocket Protocol (ws:// or wss://) over TCP

üéØ Real-World Example: User Management

Dekho har protocol mein same operation kaise hota hai:

REST Example

// Get User GET /api/users/123 Response: { "id": 123, "name": "Ali" } // Create User POST /api/users Body: { "name": "Ali", "email": "ali@test.com" } // Update User PUT /api/users/123 Body: { "name": "Ali Ahmed", "email": "ali@test.com" } // Delete User DELETE /api/users/123

<!-- Get User (Sab POST request) --> POST /UserService Body: <soap:Envelope> <soap:Body> <GetUser> <UserId>123</UserId> </GetUser> </soap:Body> </soap:Envelope> <!-- Create User (bhi POST) --> POST /UserService Body: <soap:Envelope> <soap:Body> <CreateUser> <Name>Ali</Name> <Email>ali@test.com</Email> </CreateUser> </soap:Body> </soap:Envelope>

GraphQL Example

# Get User (Query) POST /graphql { user(id: 123) { name email } } # Create User (Mutation) POST /graphql mutation { createUser(name: "Ali", email: "ali@test.com") { id name } } # Subscribe to Changes (Subscription) POST /graphql (WebSocket) subscription { userUpdated(id: 123) { name email } }

gRPC Example

// Protocol Buffer Definition service UserService { rpc GetUser(UserIdRequest) returns (User); rpc CreateUser(CreateUserRequest) returns (User); rpc UpdateUser(UpdateUserRequest) returns (User); rpc DeleteUser(UserIdRequest) returns (Empty); } // Client Code (Binary data bheji jati hai) UserService.GetUser({ userId: 123 }) // Response bhi binary, automatically decoded

üéØ Quick Decision Guide

‚úÖ REST use karo agar:

  • Simple CRUD operations chahiye
  • Public API banana hai
  • Mobile app backend
  • Caching important hai

‚úÖ GraphQL use karo agar:

  • Complex nested data chahiye
  • Mobile app mein data minimize karna ho
  • Multiple resources ek request mein

‚úÖ gRPC use karo agar:

  • Microservices architecture
  • High performance chahiye
  • Server-to-server communication

‚úÖ WebSockets use karo agar:

  • Chat application
  • Live notifications
  • Real-time dashboard

‚úÖ Webhooks use karo agar:

  • Event-based notifications
  • Payment updates
  • Third-party integrations
